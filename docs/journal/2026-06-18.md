## Technical Hurdles Solved

### Agent Configuration Struggle (Babel Version Pinning)

One of the more subtle constraints uncovered this session was frontend toolchain fragility. The single-file React frontend (`frontend/index.html`) uses `@babel/standalone` loaded via CDN — and **must stay pinned at v7**. Babel v8 introduced a breaking syntax change that causes a blank page with `SyntaxError: Unexpected token '{'` for modern JSX patterns used throughout the component tree. This constraint was documented in `CLAUDE.md` to prevent accidental unpinning in future sessions.

The same "no build step" constraint that made the project fast to iterate also made it sensitive to CDN version bumps — the agent had to be careful not to suggest upgrading or switching to a bundler.

### Global `window.fetch` Override for Auth Header Injection

Prop-drilling `accessCode` through every component that calls `fetch` would have required touching 10+ call sites. Instead, a single IIFE at the top of the script block monkey-patches `window.fetch`:

```js
(function () {
  const _raw = window.fetch.bind(window);
  window.fetch = function (url, opts = {}) {
    if (typeof url === 'string' && url.startsWith('/api/')) {
      const code = localStorage.getItem('jobAgentCode') || '';
      opts = { ...opts, headers: { 'X-Access-Code': code, ...(opts.headers || {}) } };
    }
    return _raw(url, opts).then(res => {
      if (res.status === 401) {
        localStorage.removeItem('jobAgentCode');
        window.dispatchEvent(new Event('auth-clear'));
      }
      return res;
    });
  };
})();
```

This keeps all existing fetch call sites untouched while adding auth transparently. The `auth-clear` custom event lets the `App` component subscribe and re-show the gate without polling.

### `\bword\b` vs Bare String in Regex Filter

The `_CLOSED_SIGNAL` closed-job filter required `\bclosed\b` (word-boundary anchors) rather than bare `closed`. Without them, `"Not disclosed"` (a common salary/location field value) matched and caused open jobs to be silently dropped — a data-quality bug that only surfaced when examining false-negative results.

## Architectural Insights

### Separation of Concerns: CLI vs API vs Cron

The project cleanly separates three execution modes:
- `main.py` — pure CLI (`typer`), stateless, no network dependencies
- `api.py` — FastAPI server, delegates all scan logic to `main.py` as a subprocess, manages `_state` dict with a `threading.Lock`
- `daily_scan.py` — cron entry point, calls `main.py scan` directly then `notify.py`

This means the API server never directly imports scan logic — it only shells out. This keeps the server stateless with respect to scan results (all state lives in JSON files on disk) and avoids any module-level side effects from the scan pipeline running inside the server process.

### Pending Batch Queue Pattern

The `pending_candidates.json` / `score-more` flow is a lightweight job queue without any queue infrastructure:
1. Scan scores the top 50 candidates and saves the rest to `pending_candidates.json`
2. The user can trigger `score-more` which reads the next batch, scores it, and **merges** into `last_run.json`
3. The API server exposes `pending_count` so the frontend can show how many remain

The merge step in `_score_more_pipeline()` is a known potential bug — the write path may not be atomic, so the merged count (100) could revert to the pre-merge count (50) under certain race conditions. Flagged for investigation.

### Medium/Low Security Items — Deferred to Post-Launch

These were explicitly out of scope for today:
- CORS lockdown (currently `allow_origins=["*"]`)
- Structured logging / request logging middleware
- Input length limits on chat messages and form fields
- UPLOAD_DIR periodic cleanup (stale PDFs accumulate)
- Subresource Integrity (SRI) hashes for CDN scripts

#### Human-Readable Summary

Today was mostly about making the project safe enough to put on GitHub without embarrassing myself. Before I could do a `git init`, I had to go through the code and pull out all the credentials that were just sitting there in plain text — my Gmail app password was hardcoded in `scan_config.json`, and my Anthropic API key had been pasted directly into a Claude settings file. Got those into a `.env` file, made sure git would never touch it, and only then felt okay pushing.

The other fixes were more technical but straightforward: someone could have uploaded a file named `../../api.py` and overwritten my server code (path traversal), so I replaced the user-controlled filename with a random UUID. Added an access code gate so the app isn't wide open if the port ever gets exposed. Dropped in DOMPurify so Claude's replies can't inject scripts into the page.

The painful part was the git history. I did `git add .` without thinking, and it scooped up my resume PDFs and a screenshots folder. Had to wipe the entire `.git` folder and start over from scratch — which sounds scary but is actually fine when you only have one commit. Still, it was a reminder that `git add .` is a trap when you haven't nailed down your `.gitignore` first.

The engineering lesson: **set up your `.gitignore` before you write a single line of code, not after.** Credentials and personal files get committed exactly once, and cleaning that up is always more painful than doing it right at the start.
